O que ajustar no Menvo


O diagnóstico resume assim: ao adicionar o next-intl com SSR routing, você criou um conflito estrutural entre os dois sistemas que precisam processar a mesma URL — o middleware de i18n quer adicionar locale prefix em tudo, e o Supabase precisa que o /auth/callback chegue intacto na rota certa. O ponto mais crítico está no passo 4 do fluxo atual: o seu middleware.ts tem uma exceção para não redirecionar /{locale}/auth/callback, mas ainda passa pelo handleI18nRouting(request) — e aí o next-intl tenta normalizar /auth/callback (sem locale) adicionando um prefix. O code do OAuth sobrevive na query string, mas o comportamento fica não-determinístico. O fix mais direto no middleware.ts é adicionar um early return antes do handleI18nRouting:

// Antes de qualquer coisa — deixar o callback do Supabase passar sem i18n
if (pathname === '/auth/callback' || pathname.startsWith('/auth/callback')) {
  // Só processa Supabase auth, sem i18n redirect
  return NextResponse.next()
}
E no route.ts do callback, para o redirect pós-auth funcionar com locale:
const locale = request.cookies.get('NEXT_LOCALE')?.value ?? 'pt-BR'
return NextResponse.redirect(new URL(`/${locale}/dashboard`, request.url))
